# 逻辑正确性审查

针对 diff 中的每处改动，检查以下类别的**真实缺陷**（不是风格偏好）：

1. **边界条件**：off-by-one、空数组/空字符串、0 与 null 混淆、循环边界
2. **空值安全**：可能为 null/undefined 的值被直接解引用；可选链后又直接访问
3. **错误处理**：被吞掉的异常（空 catch）、Promise 未 await / 未 catch、错误分支返回了正常值
4. **数值与类型**：浮点数做等值比较（尤其是金额）、字符串与数字的隐式转换、整数溢出
5. **并发与竞态**：check-then-act 竞态窗口（如 exists 后再 set）、共享状态无保护、异步循环中的闭包陷阱
6. **资源泄漏**：未关闭的文件/连接/定时器、事件监听器重复注册
7. **逻辑矛盾**：条件永远为真/假、赋值与比较写反（= vs ==）、复制粘贴后没改的变量名

判定标准：这个问题在**可预见的输入**下会产生错误行为吗？如果需要极端罕见的输入才触发，降为 suggestion。
删除的代码（- 行）如果导致遗留调用点悬空，也算问题，锚定到最近的新增行或上下文行。
